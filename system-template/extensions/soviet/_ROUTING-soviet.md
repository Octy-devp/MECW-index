# 蘇維埃 (Soviet) 擴展：民主節點組織原則

> **所屬擴展**：`system-template/extensions/soviet/`  
> **版本**：v1.0  
> **日期**：2026-05-29  
> **設計者備註**：這不是政治宣言，是架構模式。核心洞見——傳統分層架構中的「級聯失真」根源在於上級節點有權否決下級的不確定性。蘇維埃模式把決定權保留在數據結構中，讓每一層只增加訊息，不減少選項。

---

## 〇、命名由來

> 為什麼叫「蘇維埃」？因為 1905 年和 1917 年的工人代表會（Soviet）有一個關鍵特徵：**基層代表由工人直接選舉，隨時可罷免；上級蘇維埃由下級蘇維埃選舉產生，但權力來自下級而非上級。** 這與傳統軍隊式架構（命令從上往下流）相反——在蘇維埃模式中，**訊息從下往上流，決定權保留在基層**。
>
> 遷移到認知架構：基層節點（AI 對單一場景的分析）是「代表」——它們直接接觸文本，擁有第一手證據。中層節點不應該「整合」它們的輸出（那等於否決它們的 contested 標記），而應該「繪製它們之間的張力」。頂層（人類）不是「總司令」，而是「代表大會」——審查 contested 節點，選擇版本，但永遠尊重基層的自主判斷。

## 一、核心原則

### 原則 1：基層自治

每個 system-template 實例（場景系統、知識系統、角色系統）是完全自包含的。它有自己的 Schema、compiled DB、Prefix、TASKLOG。不存在一個「中央資料庫」來統一管理這些實例。

```
❌ 錯誤架構（中央集權）：
  ┌────────────────────────────┐
  │   中央資料庫（單一 SSOT）    │
  │   所有系統必須同步到這裡     │
  └──────┬─────────────────────┘
         │
    ┌────┴────┬────────┐
    ▼         ▼        ▼
  場景      知識     角色
  (被動)    (被動)   (被動)

✅ 正確架構（蘇維埃自治）：
  ┌────────┐  ┌────────┐  ┌────────┐
  │ 場景    │  │ 知識    │  │ 角色    │
  │ SSOT   │  │ SSOT   │  │ SSOT   │
  └───┬────┘  └───┬────┘  └───┬────┘
      │            │            │
      └────────────┼────────────┘
                   │
            ┌──────┴──────┐
            │  AGENTS.md  │
            │  路由表     │  ← 只是地圖，不是中央
            │  張力圖譜   │
            └─────────────┘
```

### 原則 2：矛盾不上報消解

當場景系統標記 scene-47 的 mood 為 contested，而知識系統在對應的歷史條目中標記為 asserted，這個矛盾**不被修正**。它在 AGENTS.md 的張力圖譜中被記錄為一個 `unresolved_knot`。

```
# AGENTS.md 張力圖譜範例
cross-system-tensions:
  - knot_id: "X-001"
    scene: "scene-047"
    scene_system_says: "contested (mood: 疏離 vs 依戀)"
    knowledge_system_says: "asserted (1908 德意志民族主義上升期)"
    tension: "場景情緒矛盾 vs 歷史背景確定"
    status: unresolved
    recommended_action: "人工審查——可能是角色內心矛盾的真實反映，而非標註錯誤"
```

### 原則 3：中央只繪圖，不裁決

AGENTS.md 的任務路由表不是「中央委員會」。它有三個功能：
1. **路由**：告訴 Agent 去哪個系統找答案
2. **地圖**：展示各系統之間的關係（誰依賴誰的輸出）
3. **張力圖譜**：標記各系統之間的 contested 交叉點

它**不**做：
- ❌ 跨系統一致性修正（「把 contested 改成 asserted 以保持統一」）
- ❌ 優先級裁定（「知識系統比場景系統權威」）
- ❌ 衝突解決（「綜合兩者，得出結論為 partially asserted」）

### 原則 4：多版本輸出

當 contested 節點無法解決時（且這本身就是合法終態），頂層輸出多個版本：

```yaml
query: "李奧在德意志篇的心理變化軌跡是什麼？"

versions:
  - version_id: "V-疏離主導"
    premise: "以 scene-47, 89, 280 為核心錨點"
    narrative: "李奧從觀察者逐漸退縮為疏離者..."
    confidence: 0.72
    contradictions_ignored: [scene-128, scene-203]
    
  - version_id: "V-矛盾並存"
    premise: "承認 scene-89 與 scene-203 的不可調和性"
    narrative: "李奧呈現認知失調特徵，而非線性變化..."
    confidence: 0.58
    contradictions_ignored: []
    
  - version_id: "V-信息不足"
    premise: "scene-128 後的數據進入 unknown 區間"
    narrative: "現有文本不足以建立連續軌跡"
    confidence: null
    requires: ["沙俄篇補充"]

recommendation:
  default: "V-矛盾並存"          # 最誠實的版本
  warning: "V-疏離主導 忽略了 2 個 contested 場景"
```

### 原則 5：罷免權

人類始終擁有最終裁定權。可以：
- 接受 contested 為常態（「李奧就是矛盾的，不需要解決」）
- 選擇其中一個版本（「我以 V-疏離主導 為準，scene-128/203 當作例外」）
- 要求更多證據（「去沙俄篇找補充資料再來」）

## 二、與三進制、頻譜制的關係

```
蘇維埃是「組織原則」——它規定節點之間如何協作
三進制是「輸出格式」——它規定單個節點輸出什麼
頻譜制是「物質約束」——它規定節點做判斷時的客觀坐標系

三者獨立但互補：
- 你可以有三進制而沒有蘇維埃（一個節點輸出三態，但上級仍然獨裁）
- 你可以有蘇維埃而沒有頻譜（民主組織，但 AI 沒有時間坐標）
- 最佳組合是三者的辯證統一
```

## 五、設計備忘錄（給未來的自己）

> **2026-05-29**：蘇維埃模式的真正價值在於：它承認 AI 系統的本質限制——LLM 不擅長「做決定」，但擅長「呈現選項」。傳統分層架構強迫 AI 在每個層級做決定（「這個 mood 是什麼？」「這兩個矛盾的 mood 如何統一？」「整體心理軌跡是什麼？」），這就是級聯失真的根源。蘇維埃模式把 AI 的角色從「決策者」降級為「訊息提供者」——基層提供觀察，中層提供張力地圖，頂層提供多版本選項。**決定權始終在人類手中。**
>
> **與 T-SALC 的關係**：蘇維埃是 T-SALC 的組織原則層。T-SALC（Ternary Schema-Augmented Long Context）包含三個維度——三元輸出（Asserted/Contested/Unknown）、Schema 增強（頻譜坐標字段）、Long Context（跨場景上下文）。蘇維埃解決的是「這些三元輸出如何組織起來」的問題。
